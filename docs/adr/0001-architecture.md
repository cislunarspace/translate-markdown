# ADR-0001：英文 Markdown 翻译工具总体架构

## 背景

需要将已整理好的英文 Markdown 文档翻译成中文。翻译后的文档应与源文档位于同一目录，以便图片等本地资源引用路径保持有效。目标用户为开发者本人，工具以 Python 实现，要求操作简单、结果可靠。

## 决策

采用单文件脚本 `main.py`，内部按职责划分为四个模块：配置模块、LLM 模块、后端模块、前端模块。

## 理由

- **单文件脚本**降低了运行和分发成本，适合个人工具。
- **模块内部职责分离**使得预处理、翻译、进度保存、界面展示可以独立演进和测试，即使所有代码在一个文件中。
- **配置持久化到本地 JSON**，避免每次启动重复输入 API Key 和请求地址。
- **Markdown 解析使用正则 + 状态机**，而非完整解析库。需求是“保护特定结构”而非“渲染 HTML”，完整解析库会引入不必要的依赖，并可能改变原始格式。
- **表格整体作为单个 TranslationUnit 发送给 LLM**，在 prompt 中要求保持 Markdown 表格语法，避免表格被物理行拆散。
- **Checkpoint 保存中间结果**，与目标文档同目录，支持断点续翻。

## 模块职责

| 模块 | 职责 | 对外接口 |
|------|------|---------|
| 配置模块 | 读写本地 JSON 配置 | `load_config()` / `save_config()` |
| LLM 模块 | 调用 DeepSeek API，处理重试、限流 | `LLMClient.translate_batch(units)` |
| 后端模块 | 预处理、翻译循环、断点续翻、合并输出 | `translate_document(source, config, sink)` |
| 前端模块 | PyQt6 GUI 与命令行入口 | `run_gui()` / `run_cli()` |

## 关键数据结构

- `Block`：保护块或待译块，是后处理合并的基本单元。
- `TranslationUnit`：发送给 LLM 的最小翻译单元。
- `TranslationResult`：LLM 返回的翻译结果。
- `Checkpoint`：断点续翻的持久化状态，包含源文档 hash、完整块列表和已完成 unit_id。

## 主流程

1. 读取源文档并计算 MD5。
2. 预处理：识别保护块（代码块、表格、图片路径），生成 `Block` 和 `TranslationUnit` 列表。
3. 检查 checkpoint：若存在且源文档 hash 匹配，则恢复进度；否则从头开始。
4. 翻译循环：逐批调用 LLM，回填结果，更新 checkpoint，汇报进度。
5. 后处理：合并 blocks，重组为完整 Markdown。
6. 写入目标文档 `{stem}_zh.md`。
7. 删除 checkpoint，完成最终汇报。

## 风险与缓解

- **代码块边界误判**：使用行首匹配的状态机，记录起始标记，避免被代码块内部的反引号干扰。
- **表格中的行内代码**：预处理按“代码块 → 表格 → 行内元素”的顺序识别，行内代码在生成 TranslationUnit 时替换为占位符。
- **LLM 失败或限流**：实现指数退避重试，批量失败时降级为单条请求。
- **源文档变更后误用旧 checkpoint**：checkpoint 保存 `source_hash`，恢复时校验，不匹配则重新开始。
